# Conductor Context and Auto-Init Design

**Date:** 2026-02-25  
**Branch:** multipowers

## Goal
Replace `.claude/session-*` spec artifacts with Conductor-style project context under `conductor/`, and enforce context-driven execution for all spec-driven `/mp` commands.

## Scope
- Add `/mp:init` interactive setup flow (Conductor-style).
- Store templates in `custom/templates/`.
- Generate project context artifacts under target project's `conductor/`.
- Add command-level guard for spec-driven commands to auto-run init if context is missing/incomplete.
- Move spec tracking to `conductor/tracks/` with checkbox task state.

## Non-Goals
- No migration of unrelated command families.
- No requirement to change non-spec-driven commands.
- No heavy rewrite of entire orchestrator routing model.

## Decisions
1. **Guard level:** Command-level guard (not global script gate).
2. **Init behavior:** Auto-run interactive `/mp:init` when spec-driven command needs context and it does not exist.
3. **Context location:** `conductor/` in target project root.
4. **Spec tracking:** `conductor/tracks/*.md` with checkbox status.
5. **Plan/intent location:** use `conductor/tracks/<track_id>/plan.md` and `conductor/tracks/<track_id>/intent.md` for spec-driven flows.

## Target Artifacts
Generated by `/mp:init` into the target project:
- `conductor/product.md`
- `conductor/product-guidelines.md`
- `conductor/tech-stack.md`
- `conductor/workflow.md`
- `conductor/code_styleguides/`
- `conductor/tracks.md`

Additional runtime artifacts:
- `conductor/tracks/YYYY-MM-DD-<slug>.md`

## Architecture
### 1) Conductor Context Module
Add a reusable context module under custom overlay libs, e.g. `custom/lib/conductor-context.sh`:
- `is_spec_driven_command <command>`
- `conductor_root_exists`
- `conductor_context_complete`
- `ensure_conductor_context <command>`
- `load_conductor_context_for_prompt`

### 2) Init Command
Create `/mp:init` command and corresponding orchestrator entry:
- Interactive Q&A setup flow borrowing Conductor setup UX.
- Writes templated files from `custom/templates/`.
- Per-file overwrite decision (`keep`, `update`, `replace`).

### 3) Template Source of Truth
Store maintained templates at:
- `custom/templates/conductor/*.md`
- `custom/templates/conductor/code_styleguides/*`

### 4) Spec-Driven Command Guard
For these commands, call `ensure_conductor_context` before execution:
- `/mp`
- `/mp:plan`
- `/mp:discover`
- `/mp:define`
- `/mp:develop`
- `/mp:deliver`
- `/mp:embrace`
- `/mp:review`
- `/mp:debate`
- `/mp:research`

### 5) Spec Tracking Model
For each spec-driven run:
- Ensure a track file in `conductor/tracks/`.
- Use checkboxes (`- [ ]`, `- [x]`) for tasks/subtasks and validation gates.
- Update status as command progresses.

## Data Flow
1. User runs a spec-driven command.
2. Guard checks context completeness.
3. If missing: auto-run `/mp:init` interactive setup.
4. Guard re-checks context.
5. Command loads context docs from `conductor/`.
6. Command creates/updates `conductor/tracks/<run>.md` checkbox plan.
7. Command executes workflow and updates completion status.

## Error Handling
- If init is canceled or context still incomplete: fail command with actionable guidance.
- If template render fails: stop and show missing variable/file details.
- If track update fails: continue command but emit warning and fallback to stdout status.

## Compatibility
- Keep existing non-spec commands unchanged.
- Preserve existing command names and user-facing behavior, except context storage location and auto-init flow.
- Remove `.claude/session-*` writes from spec-driven commands.

## Validation Strategy
- Unit tests for context completeness checks and spec-driven allowlist.
- Integration tests for:
  - missing context -> auto init
  - context read before execution
  - track file creation and checkbox updates
  - no `.claude/session-*` writes for spec-driven commands

## Open Risks
- Upstream churn in `.claude/commands/*` may cause frequent merge conflicts.
- Interactive init in non-interactive environments needs fallback behavior.

## Mitigation
- Keep logic centralized in `custom/lib/*` and thin command hooks.
- Add clear test contracts to detect regressions early.
